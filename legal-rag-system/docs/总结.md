# 法律 RAG 多智能体协作系统

---

## 一、项目概述

基于 DeepSeek API 构建的法律智能助手，集成三种推理范式（ReAct / Plan-Solve / Reflection），支持多层级法律知识库检索与流式对话交互。

---

## 二、技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | Vanilla JS + SSE + Marked.js | 零框架依赖，纯原生实现 |
| 后端框架 | FastAPI + StreamingResponse | 异步高性能，原生 SSE 支持 |
| 向量数据库 | ChromaDB (PersistentClient) | 本地持久化，ONNX 模型推理 |
| Embedding | ONNX all-MiniLM-L6-v2 | 轻量级多语言嵌入模型（~90MB） |
| LLM | DeepSeek API（OpenAI 兼容） | 国产大模型，法律中文理解优秀 |
| 容器化 | Docker + docker-compose | 一键构建部署，数据卷持久化 |
| 文档解析 | python-docx + PyPDF2 | 支持 .docx / .pdf / .txt / .md / .json |
| 天气服务 | wttr.in | 免费、无需 API Key 的天气查询 |
| 工具协议 | MCP (Model Context Protocol) | 标准化工具发现与调用，SSE over HTTP 传输 |

## 五、知识库架构创新

### 1. 三层知识库设计

构建「法规库 — 案例库 — 实务库」三层向量知识库，每层赋予不同权威权重：

| 层级 | 权威权重 | 内容 | 分块策略 |
|------|---------|------|---------|
| 法规库 | 1.0 | 民法典、劳动法等正式法律文本 | 按「第X条」条文拆分 |
| 案例库 | 0.6 | 裁判文书、指导性案例 | 按章节标题拆分（裁判要旨/基本案情/裁判理由等） |
| 实务库 | 0.4 | 合同问答、操作指引、文书模板 | 先试条文拆分，回退段落+强制定长切分 |

**创新点**：权威权重参与相关度评分和置信度计算，使法规引用的法律效力天然高于案例和实务资料。

### 2. 混合检索引擎

结合向量语义检索与中文 n-gram 关键词匹配，解决通用 embedding 模型对中文法律文本理解不足的问题：

```
相关度 = 关键词匹配(40%) + 来源权威度(30%) + 向量距离(30%)
```

- 2-gram / 3-gram 中文分词级匹配，无需依赖分词器
- 全文精确匹配额外加分（+10）
- 关键词权重 70% + 向量权重 30% 的混合重排序

**创新点**：绕过 embedding 模型的中文法律语义瓶颈，精确命中具体法条编号（如「第1084条」）。

### 3. 同源频率加成

同一文档的多个分块命中查询时，所有来自该文档的引用获得相关度加成：

```
加成系数 = min(1.3, 1.0 + 0.05 × (命中块数 - 1))
```

- 1 块命中 → 无加成
- 4 块命中 → +15%
- 7+ 块命中 → +30%（封顶）

**创新点**：文档级信号增强，避免分块后同源引用分散导致相关度被低估。

## 六、置信度评分与结构化引用

### 置信度计算（v2：基于引用质量）

置信度改为基于引用来源相关度计算，封顶 92%：

```
confidence = 去掉最大最小值后的引用相关度平均值
```

- 引用 ≥ 3 条时去极值取平均，抗极端值干扰
- 引用 1-2 条时直接取平均
- 无引用回退 0.25
- 下限 20%，上限 92%，永不为 100%

### 结构化引用提取

引用来源由三部分合并，按相关度降序排列：

| 来源 | 方法 | 说明 |
|------|------|------|
| KB 检索 | `_extract_citations` | 回答实体交叉匹配，实际引用的获 2x 加成 |
| 联网 URL | `_extract_web_urls` | 从回答文本提取真实 URL，上下文推断标题 |
| 回答法条 | `_extract_answer_statute_refs` | 正则匹配 `《法》第X条`，仅在 KB/联网有结果时启用 |

每条引用附带：来源类型、标题、内容摘要、相关度评分（封顶 90%）、条文编号。

## 七、流式推理架构

### 1. SSE 流式输出

前端通过 `fetch + ReadableStream` 消费 POST 端点的 SSE 事件流，实现三种事件类型：

| 事件 | 作用 | 渲染方式 |
|------|------|---------|
| `thinking` | 推理中间步骤 | 逐字打字机（30ms/字），颜色编码 |
| `token` | 最终答案 token | 逐字打字机（10ms/字），Markdown 实时渲染 |
| `done` | 元数据 | 置信度 + 引用来源 + 推理范式标签 |

**创新点**：POST 端点流式输出而非传统 GET SSE，支持携带完整请求体（推理模式、性格配置、会话 ID）。

### 2. 三种推理范式的流式实现

**ReAct 模式**：Thought → Action → Observation 循环可视化为思考步骤，LLM 输出 3 字符缓冲流式渲染，工具调用结果分块（80 字/块）避免前端卡顿。

**Plan-Solve 模式**：规划 → 逐步执行 → 综合合成，每个阶段的中间结果作为 thinking 事件输出，最终答案通过 LLM 流式生成。

**Reflection 模式**：初始回答 → 反思审查 → 改进迭代 → 最终答案，每次反思反馈和改进结果均可视化。

**创新点**：用户在等待时可以观察智能体的「思考过程」而非空白等待，提升信任感和可解释性。

### 3. 逐字打字机效果

采用 `setInterval`（10ms 间隔）替代 `requestAnimationFrame`，实现平滑逐字输出：

- 流式阶段：每 10ms 输出 1 字符
- 完成阶段：自动加速为每 10ms 输出 5 字符追赶到末尾
- 不会出现「突然蹦出一大段」的问题

### 4. 即时反馈设计

用户提交问题后立即显示「正在分析您的问题...」→「正在检索知识库...」→「检索完毕，启动推理引擎...」，消除长时间白屏等待。

## 八、Agent 自主工具选择与决策

### Agent 的核心理念

传统程序通过硬编码的 `if-else` 逻辑决定调用哪个函数。而智能体（Agent）的本质是：**LLM 作为「大脑」，自主感知、推理并决定何时使用何种工具，而非由程序预设路径**。

本系统中，ReAct 智能体完整实现了这一范式：

```
感知（Perception）      推理（Reasoning）        行动（Action）
     │                       │                      │
     ▼                       ▼                      ▼
┌──────────┐    ┌──────────────────────┐    ┌──────────────┐
│ 可用工具列表 │ → │ LLM 自主思考需要什么信息 │ → │ 自主选择工具+参数 │
│ 执行历史    │    │ 工具A能做什么？够不够？  │    │ 执行 → 观察结果   │
│ Observation │    │ 是否需要再查另一个工具？ │    │ 循环或输出答案   │
└──────────┘    └──────────────────────┘    └──────────────┘
```

### 如何实现自主工具选择

**Step 1 — 注入工具清单**：将工具名和功能描述注入 System Prompt，LLM 通过语义理解每个工具的用途，而非代码层面的函数签名匹配：

```
## 可用工具
- search_statutes: 检索相关法律法规，输入法条名称或关键词
- search_cases: 检索相关案例，输入案由或关键词
- check_validity: 检查法规时效性，输入法规名称
- related_laws: 查找与给定法规相关的上位法、下位法、特别法
- calc_litigation_fee: 诉讼费计算器
- statute_limitation: 诉讼时效查询
- get_time_weather: 获取当前日期时间和当地天气信息
```

**Step 2 — LLM 自主决策**：没有任何代码规定「劳动争议问题→调用 search_statutes」。LLM 根据问题内容、工具描述、执行历史自主输出 Thought 和 Action：

```
Thought: 用户询问劳动仲裁材料，需要检索相关法规和实务指南
Action: search_statutes[劳动仲裁 申请材料]
```

**Step 3 — 反馈循环**：工具执行结果（Observation）回传给 LLM，LLM 判断信息是否充足，自主决定继续调用工具还是输出最终答案：

```
Observation: [法规检索结果] 共找到3条...（包含《劳动争议调解仲裁法》第28条等）

Thought: 已获得足够的法条依据，可以给出完整回答
Action: Finish[根据《劳动争议调解仲裁法》第28条，申请劳动仲裁需要...]
```

### 完整执行实录

以用户问题「公司拖欠工资，如何申请劳动仲裁？」为例，ReAct 智能体的真实推理过程：

```
Step 1:
  Thought: 用户咨询拖欠工资的劳动仲裁流程。需要同时检索相关法条和案例，
           了解申请条件和所需材料。
  Action:  search_statutes[拖欠工资 劳动仲裁 申请条件]

  Observation: [返回《劳动争议调解仲裁法》第2条、第27条、
               《劳动合同法》第85条相关内容...]

Step 2:
  Thought: 已获得法条依据。还需要查找具体操作流程和仲裁申请书模板等
           实务资料，也看看有没有相关案例参考。
  Action:  related_laws[劳动仲裁]

  Observation: [返回《劳动争议调解仲裁法》《劳动法》《劳动合同法》
               相关条文...]

Step 3:
  Thought: 检索结果已覆盖法条、案例和实务指南，信息充足。可以综合给出
           完整回答：先说明法律依据，再列出申请步骤和所需材料，最后引用
           相关法条。
  Action:  Finish[根据检索结果，申请劳动仲裁的主要流程和材料如下：...]
```

### 自主决策的关键特征

| 特征 | 传统程序 | Agent（本系统） |
|------|---------|---------------|
| 工具选择 | 代码写死 `if intent=="labor" → search_statutes` | LLM 阅读工具描述后自主决定 |
| 调用时机 | 预设执行顺序 | LLM 根据 Observation 动态判断是否需要更多工具 |
| 参数构造 | 固定模板拼接 | LLM 根据语境自由组织检索关键词 |
| 多工具组合 | 需预定义每个组合路径 | LLM 自主组合 search_statutes + related_laws + search_cases |
| 终止条件 | 执行完所有步骤 | LLM 自主判断信息是否充足，主动发出 Finish |
| 简单问题 | 仍然走全流程 | LLM 识别为问候/闲聊，跳过工具直接 Finish |

### 为什么这是「Agent 思想」而非「if-else 路由」

**核心区别在于决策权归属**：

- **if-else 路由**：代码决定 → `意图识别模块` → `根据意图查表选工具` → `调用工具`。每个分支是程序员预先编码的。
- **Agent 自主选择**：LLM 决定 → 工具描述只是「菜单」，LLM 是「点菜的人」。同一个问题，不同 LLM 或不同上下文可能选择不同工具路径。

本系统的 ReAct Agent 没有一行代码说「如果是劳动争议就调 search_statutes」。LLM 在 prompt 约束下，**基于语义理解自主判断**该调哪个工具、传什么参数、是否需要继续——这才是 Agent 的本质。

### 四层联网搜索框架 — 结构化 Web 信息获取

本地知识库在时效性上天然滞后（如无法覆盖最新修订的公司法）。系统实现了四层结构化联网搜索框架，Agent 自主完成「决策→解析→搜索→抓取→合成」全流程，将互联网实时信息融入法律推理。

**第一层：决策 — 要不要搜**

Agent 在调用任何联网工具前首先判断必要性：
- 纯问候/闲聊（"你好""谢谢"）→ 不搜索，直接 Finish
- **其他一切问题**（法律+非法律）→ 必须搜索，即使"知道"答案也要用工具验证
- KB 结果与问题主题不符 → 联网搜索，不等不靠
- 拿不准就搜，不依赖模型训练数据

**第二层：解析 — 搜什么**

将自然语言问题转化为精准搜索关键词：
- 提取核心实体和关键概念（法律名称、案由、事实要素）
- 去掉疑问词、语气词、停用词
- 生成 2-3 组不同粒度的搜索词，提高命中率

**第三层：执行 — 怎么搜**

```
Step 1: mcp_search_web[关键词] 获取搜索结果列表
Step 2: 审视摘要，选 1-3 条最相关的
Step 3: mcp_fetch[选中URL] 深度抓取全文
Step 4: 抓取完成 → 直接进入合成，禁止重复搜索
约束：同一关键词最多搜 1 次，联网最多 2 次不同关键词，2 次无结果立即 Finish
```

**第四层：合成 — 怎么答**

综合工具返回的真实内容回答，不得插入模型训练数据。每一条事实性陈述必须有 Observation 来源。最终回答严禁出现「据我所知」「此信息来源于训练数据」等措辞。每个法律结论标注来源，区分「明确法律规定」与「实践倾向」。

### 多源搜索引擎 — `mcp_search_web` 工具

新增 `mcp_search_web` MCP 工具，实现多引擎级联搜索：

```
查询 → Wikipedia(zh) → Wikipedia(en) → Bing → DuckDuckGo
         ↑ 命中返回     ↑ 命中返回     ↑ JS渲染受限  ↑ 验证码问题
```

**关键词自动简化**：当长查询无结果时，自动去除修饰词提取核心关键词回退重试，如 「公司法 注册资本 2025 修订」→「公司法」。

| 引擎 | 方式 | 状态 | 说明 |
|------|------|------|------|
| Wikipedia (zh/en) | OpenSearch API | ✅ 主力 | 标题+摘要+URL，免费无限制 |
| Bing | HTML 解析 | ⚠️ 受限 | JS 动态渲染，容器 IP 可能返回空页 |
| DuckDuckGo Lite | HTML 解析 | ⚠️ 备用 | 服务器 IP 可能触发验证码 |

2. 四层决策框架内嵌到 prompt，LLM 自主判断每层走向

### 反训练数据污染 — 三重防护体系

系统通过提示词 + 代码 + 流程控制构建了三层防护，确保不输出预训练知识：

| 层级 | 机制 | 说明 |
|------|------|------|
| **提示词** | 核心铁律 × 4 处 | 禁止使用训练数据、禁止说"据我所知"、失败只能给建议、附声明也不行 |
| **代码** | 关键词拦截 | `训练数据` `据我所知` `我记得` `预训练` `我的知识库` 命中立即替换 |
| **流程** | Finish 直接返回 | Agent 的 Finish 跳过合成路径，不被空 Observation 的合成覆盖 |

### 智能循环终止

- 工具返回「已被禁用」→ 立即 Finish，不重试
- 搜索无结果连续 2 次 → 自动注入 Finish 强制停止
- 代码级重复检测覆盖 `未找到` `无结果` `失败` `已被禁用` `disabled`

### 三种范式统一提示词体系

ReAct / Plan-Solve / Reflection 共享同一套核心铁律：

- 禁止使用模型训练数据（含附声明输出）
- 非法律问题也必须搜索
- 参数格式 `key=value`（非 `key: value`）
- 联网最多 2 次不同关键词，无结果立即 Finish
- Observation 有数据就必须回答，禁止说找不到
- Thought 只描述需要查什么，不提"知道"什么

## 九、MCP 协议集成 — 进阶工具发现与调用

### 什么是 MCP

MCP（Model Context Protocol）是 Anthropic 提出的标准化 AI 工具协议，用于统一 LLM 与外部工具/数据源的通信方式。传统工具注册方式下，每个工具函数的实现、参数映射、错误处理都与系统紧耦合。MCP 将其解耦为独立的「工具服务端」和「工具客户端」，实现：

- **工具提供方与使用方分离**：工具实现者只需关注 MCP 服务端，不关心谁来调用
- **零侵入扩展**：新增工具不需要修改主系统代码，重启 MCP 服务端即可
- **跨语言互操作**：MCP 服务端可以用任何语言实现，通过标准协议与主系统通信

### 系统架构

### 传输层选择：从 stdio 到 SSE

MCP 协议支持多种传输方式。最初选择 stdio（标准输入输出）作为通信管道：

```
主进程 ←→ stdio ←→ MCP 服务端子进程
```

**遇到的问题**：MCP Python SDK 内部使用 `anyio` 库管理异步任务。当 stdio 客户端在后台线程的事件循环中运行时，`anyio` 的 cancel scope 跨任务管理机制触发 `RuntimeError: Attempted to exit cancel scope in a different task than it was entered in`。这是 `anyio` + `asyncio` 多线程混用的已知限制。

**解决方案 — SSE over HTTP**：将 MCP 服务端改为 HTTP 服务（FastMCP + Uvicorn），通过 SSE（Server-Sent Events）进行双向通信：

- 服务端：`mcp.sse_app()` 返回 Starlette 应用，Uvicorn 在 `127.0.0.1:8765` 提供服务
- 客户端：`mcp.client.sse.sse_client()` 通过 HTTP/SSE 连接，绕过 stdio 的 anyio 线程问题
- 生命周期：MCP 服务端作为独立子进程启动，MCP 客户端启动时重试连接（5 次，间隔 1 秒）

**[app/tools/mcp_client.py](app/tools/mcp_client.py)** — MCP 客户端管理器（~364 行）：
- `MCPClientManager`：管理多 MCP 服务端的完整生命周期（启动子进程 → SSE 连接 → 工具发现 → 持久会话）
- `MCPTool`：MCP 远程工具的本地代理，实现 `execute()` 接口供 ToolRegistry 统一调用
- `call_tool_sync()`：通过 `asyncio.run_coroutine_threadsafe()` 实现 async→sync 桥接
- `add_local_server()`：支持 `module` 参数指定不同服务端模块，实现多服务端并存

**关键设计决策 — `remote_name` 字段**：

MCP 工具注册到 ToolRegistry 时加 `mcp_` 前缀避免与本地工具重名（如 `search_all` vs `mcp_search_all`），但 MCP 服务端的实际工具名不带此前缀。早期版本在 `MCPTool.execute()` 中误用注册名（如 `mcp_fetch`）调用服务端，导致 `"Unknown tool: mcp_fetch"` 错误。修复方案：新增 `remote_name` 字段存储服务端原始工具名，`execute()` 时使用 `remote_name` 调用服务端。

```python
class MCPTool:
    def __init__(self, ..., remote_name: str = None):
        self.name = name                    # 注册名: "mcp_fetch"
        self.remote_name = remote_name or name  # 服务端名: "fetch"

    def execute(self, input_text: str = "", **kwargs) -> str:
        # ...参数映射（JSON | key=value | 裸字符串 三级解析）...
        return self._manager.call_tool_sync(
            self.server_name,    # "local-fetch"
            self.remote_name,    # "fetch" ← 使用服务端原始名称
```

## 十二、踩坑记录与解决方案

本节记录开发四层联网搜索框架过程中遇到的关键问题及解决思路，供后续维护参考。

### 1. MCP 工具名前缀不匹配

**现象**：Agent 调用 `search_web[...]` 或 `fetch[...]` 时报 `Unknown tool`，但 MCP 服务端日志显示工具已注册。

**排查**：MCP 工具注册到 ToolRegistry 时自动加 `mcp_` 前缀避免与本地工具重名（`mcp_fetch`），但 ReAct prompt 中引用的是不带前缀的名称（`search_web`、`mcp_fetch`）。两者不一致导致 Agent 调用的工具名在注册表中找不到。

**根因**：`MCPTool` 的 `name` 字段（注册名）与 prompt 中使用的名称不同步。

**解决**：
- 统一在 prompt 中使用 `mcp_` 前缀的完整名称（`mcp_search_web`、`mcp_fetch`）
- 在 [react_agent.py](app/agents/react_agent.py) 和 [orchestrator.py](app/agents/orchestrator.py) 中全局替换引用

### 2. `.strip("\"'")` 破坏 Key=Value 语法

**现象**：Agent 正确生成了 `mcp_search_web[query="关键词"]`，但搜索结果总是「Wikipedia 'query:'」，搜索词变成了字面量 `query:`。

**排查**：追踪 `_parse_tool → execute_tool → MCPTool.execute` 的完整数据流。在 `MCPTool.execute()` 中，`raw_input.strip("\"'")` 将 `query="关键词"` 的尾部引号也去掉了，变成 `query="关键词`，导致后续正则匹配失败。

**根因**：`.strip("\"'")` 是无差别去除首尾字符，不区分「包裹整个值的引号」和「key=value 语法中的引号」。`'query="xxx"'` 末尾的 `"` 是 key=value 语法的闭合引号，不应被 strip。

**解决**：改为仅 strip 成对匹配的引号（首尾字符相同且在 `'"` 中）：
```python
# Before (bug)
raw_input = raw_input.strip().strip("\"'")

# After (fix)
if len(raw_input) >= 2 and ((raw_input[0] == raw_input[-1] == '"') or ...):
    raw_input = raw_input[1:-1]
```

### 3. 正则 `(\S+)` 把逗号当值的一部分

**现象**：Agent 生成 `mcp_search_web[query=Trump, max_results=5]`，解析后 query 变成 `Trump,`（带逗号），而非 `Trump`。

**排查**：key=value 正则 `(\w+)\s*=\s*(\S+)` 中 `\S+` 匹配所有非空白字符，包括逗号。`query=Trump,` 被完整匹配为 key=query, val=Trump,。

**根因**：LLM 不总是给值加引号（`query="Trump"`），无引号时逗号分隔符被当作值的一部分。

**解决**：重写为两阶段解析器：
1. 先尝试匹配引号值：`key="value"` 或 `key='value'`
2. 无引号时按 `, key=` 边界切分：`key=value1, key2=value2`

```python
# 第一阶段：匹配引号值
m = re.match(r'(\w+)\s*=\s*(?:"([^"]*)"|\'([^\']*)\')', remaining)
# 第二阶段：按参数边界切分无引号值
m = re.match(r'(\w+)\s*=\s*(.+?)(?=$|,\s*\w+\s*=)', remaining)
```

### 4. 无引号多词值被空格截断

**现象**：`query=Donald Trump` 解析后 query 只有 `Donald`，`Trump` 丢失。

**根因**：早期修复中的 `([^,\s]+)` 同时排除了逗号和空格，导致多词查询值在空格处截断。

**解决**：上述两阶段解析器中，第二阶段使用 `(.+?)(?=$|,\s*\w+\s*=)` 匹配，只有在遇到 `, key=` 模式或字符串末尾时才停止，允许值中包含空格。

### 5. Bing 搜索返回空结果

**现象**：`_search_bing()` 请求 `https://www.bing.com/search?q=关键词` 返回 200 OK 和 69KB HTML，但从 HTML 中提取不到任何搜索结果（无 `<li class="b_algo">`、无 `<cite>`、无结果 `<a>` 标签）。

**排查**：打印原始 HTML 后发现，Bing 的搜索结果完全通过 JavaScript 动态渲染。urllib 获取的初始 HTML 只包含 `<script>` 配置块（`_G={...}`），不包含任何搜索结果 DOM。

**根因**：Bing 依赖 JS 渲染，urllib 无法执行 JavaScript。且容器 IP 可能被 Bing 识别为机器人流量。

**解决**：
- `_search_bing()` 保留但降级为备用源
- 搜索优先级调整为：Wikipedia API（主力）→ Bing（备用）→ DuckDuckGo（最后）
- Wikipedia OpenSearch API 纯 HTTP JSON 接口，无需 JS，不受 IP 限制

### 6. Wikipedia 长查询无结果

**现象**：`_search_wikipedia("公司法 注册资本 2025 修订")` 返回空，但 `_search_wikipedia("公司法")` 返回正常结果。

**根因**：Wikipedia OpenSearch API 做的是标题前缀匹配，不是全文搜索。多词组合查询很难命中精确标题。

**解决**：在 `_search_web()` 中添加关键词自动简化回退：
```python
stop_words = ['最新', '规定', '修订', '解读', '如何', '什么', '请问']
parts = [p for p in keywords.split() if p not in stop_words]
queries_to_try = [query, ' '.join(parts[:3]), parts[0]]  # 逐步简化
```

### 7. 知识库返回无关内容时 Agent 拒绝搜索

**现象**：用户问「搜索介绍 Donald Trump」，知识库返回了《监狱法》等完全不相关的内容，但 Agent 认为「已有检索结果」而跳过联网搜索，最终回答「抱歉，无法基于现有资料回答」。

**根因**：ReAct prompt 中决策层规则为「若本地知识库检索结果已充分覆盖用户问题 → 不搜索」。Agent 将「有结果返回」等同于「结果充分覆盖」，未评估结果的实际相关性。

**解决**：重写决策层 prompt，增加相关性判断：
```
- **关键判断**：先审视「法律知识检索结果」是否真正与用户问题相关。
  如果检索结果的主题与用户问题完全不同（如用户问公司法却返回监狱法），
  视为**检索无效**，必须主动触发联网搜索
- **默认原则：拿不准时就搜。宁可多搜一次，不可用无关内容敷衍**
```

### 8. LLM 训练偏见覆盖工具执行结果

**现象**：Agent 在思考过程中成功调用了 `mcp_search_web` 并获取了真实 URL（如 `https://zh.wikipedia.org/wiki/唐纳德·特朗普`），但最终回答却说「抱歉，我无法执行联网搜索功能」。

**根因**：LLM（DeepSeek）的训练数据中包含大量「我无法浏览互联网」的拒绝模板。当进入最终答案合成阶段时，模型训练偏见压倒了工具返回的实际 Observation 内容，模型「忘记」了自己已经成功获取了网络信息。

**解决**：在 3 处关键 prompt 中强制注入反偏见指令：
- **ReAct 模板**：「严禁说『我无法联网』『我无法搜索』——这些工具就是你访问互联网的手段」
- **合成 prompt**：「**重要**：以上推理过程中 Observation 部分是你通过工具获取到的真实互联网内容。严禁说『我无法联网』——你已经通过工具成功获取了信息」
- **编排器 prompt**：「**你是具备真实联网能力的智能体**」

### 9. DuckDuckGo 触发验证码

**现象**：容器 IP 请求 DuckDuckGo Lite 时频繁返回 CAPTCHA 页面，搜索不可用。

**根因**：DuckDuckGo 对数据中心 IP 有严格的速率限制和机器人检测。

**解决**：
- DuckDuckGo 降级为最后备用源
- 添加 CAPTCHA 检测逻辑（`if "captcha" in html.lower()`），检测到后静默回退
- Wikipedia API 作为主搜索源，不需要反爬处理

### 10. Wikipedia OpenSearch 返回无关结果

**现象**：搜索中文关键词时 Wikipedia 返回 `Quercus bicolor`、`Queen (乐队)`、`Query string` 等完全无关的英文条目。

**根因**：Wikipedia OpenSearch API 是标题前缀匹配，短词/英语词会匹配到大量不相关内容。原始代码不做任何结果过滤。

**解决**：在 `_search_wikipedia()` 中添加相关性过滤——从查询词提取 ≥2 字关键词，标题或摘要至少命中 1 个才保留。过滤后为 0 则返回空，上层自动切换到 Bing/DDG。

### 11. 模型在最终回答中输出训练数据

**现象**：工具搜索无结果，Agent 在 Finish 中擅自输出完整法条分析并附「此信息来源于我的训练数据」声明。

**根因**：提示词未明确禁止「附声明输出训练数据」。LLM 认为加声明就能免责。

**解决**：
- 核心铁律新增：**最终回答中严禁出现任何训练数据内容，附免责声明也不行**
- 代码级防护：检测 `训练数据` `据我所知` `我记得` `预训练` `我的知识库`，命中直接替换为安全文案

### 12. 合成路径覆盖 Agent 安全 Finish

**现象**：Agent 正确 Finish `未找到，建议查阅...`，但最终回答变为完整法律分析（包含法条、司法解释）。

**根因**：stream_run 中 Agent Finish 后仅 `break`，但 `used_tools=True` 触发后处理合成路径，合成 LLM 从空 Observation 自行生成答案，覆盖了安全 Finish。

**解决**：Agent Finish 后直接流式输出并 `return`，完全绕过合成路径，不再被覆盖。

### 13. 联网禁用时 Agent 重复尝试 8 次

**现象**：联网已禁用，Agent 连续调用 `mcp_search_web` 8 次，每次收到「已被禁用」仍不放弃。

**根因**：循环检测仅覆盖 `未找到`/`失败`，未检测 `已被禁用`。

**解决**：`mcp_search_web` 返回 `已被禁用`/`disabled` → 立即 Finish，不等第 2 次。提示词同步加入联网 2 次上限。

### 14. 参数 `key: value` 冒号格式不被识别

**现象**：LLM 写 `mcp_search_web[query: 民法典, max_results: 5]`，解析器只认 `=`，整串作为 query 文本传入搜索。

**根因**：LLM 习惯用冒号，但解析器 `"=" in raw_input` 限定等号。

**解决**：解析器同时支持 `=` 和 `:` 分隔符。提示词明确 `工具名[key1=value1, key2=value2]`，必须用 `=`。

### 15. `re.escape` 变量名错误导致搜索崩溃

**现象**：`mcp_search_web` 执行报 `name 're' is not defined`，所有搜索失败。

**根因**：解析器 `import re as _re_mcp` 但动态拼接正则时写了 `re.escape(sep)` 而非 `_re_mcp.escape(sep)`。

**解决**：2 处 `re.escape` 全部改为 `_re_mcp.escape`。

### 16. 引用来源与回答内容脱节

**现象**：回答中引用了《民法典》第一千零七十七条全文，引用列表中却没有；列表中有些来源回答并未提及。知识库空+联网禁用时仍出现引用。

**根因**：引用提取三个来源互不交叉，KB 只看查询相关度不看回答内容，法条从回答提取但未做可靠性校验。

**解决**：
- 新增 `_extract_answer_entities` + `_answer_citation_boost`：回答实际引用的来源获 2x 加成
- 新增 `_extract_answer_statute_refs`：从回答匹配 `《法》第X条` 直接生成引用
- KB+联网均无结果时跳过法条提取，不生成不可靠引用
- 合并排序按相关度降序，置信度改为引用相关度的去极值平均值，上限 90%

## 十三、总结

本系统在传统 RAG 管道之上进行了多项针对性创新：

1. **法律场景专属分块** — 按法条/章节拆分而非通用定长切分
2. **混合检索 + 同源加成** — 弥补通用 embedding 的中文法律语义短板
3. **基于引用质量的置信度** — 取引用相关度去掉极值后的平均值，直接反映信息来源可信度（封顶 92%）
4. **流式思考可视化** — 将智能体推理过程从黑盒变为可观察的对话流
5. **Agent 自主工具选择** — LLM 自主感知、推理、决策，而非硬编码 if-else 路由
6. **四层联网搜索框架** — 决策→解析→搜索→抓取→合成，Agent 自主判断搜索需求、构造关键词、多引擎级联
7. **多源搜索引擎 + 相关性过滤** — Wikipedia API + Bing + DuckDuckGo 级联，关键词自动简化回退，搜索结果关键词过滤去噪
8. **Web URL 引用提取** — 回答中真实 URL 自动提取为结构化引用，与本地知识库引用统一展示
9. **MCP 双服务端架构** — legal-tools（知识库检索）+ local-fetch（抓取+搜索），独立子进程多端口部署
10. **MCP 工具代理层** — 支持 `key=value` 和 `key: value` 双格式参数解析，适配 LLM 不同输出风格
11. **细粒度交互打磨** — 逐字打字机、进度条防超 100%、处理态锁定、思考过程自动折叠等
12. **回答感知的引用匹配** — 从回答提取法条/案例名与检索结果交叉比对，回答实际引用的来源获 2x 加成优先展示
13. **回答法条直接提取** — 正则匹配 `《法》第X条` 自动生成引用，填补 KB 和联网之外的引用空白
14. **多层级反训练数据污染** — 提示词 4 处铁律 + 代码级关键词拦截 + Finish 路径直接返回，三重防护确保不输出预训练知识
15. **智能循环终止** — 工具禁用立即停止、搜索无结果 2 次限制、合成路径不再覆盖 Agent 安全 Finish
16. **三种范式统一提示词体系** — ReAct / Plan-Solve / Reflection 共享同一套核心铁律和搜索规则
